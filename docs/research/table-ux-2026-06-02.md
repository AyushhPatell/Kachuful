





# Research: One-table flow for round end (no new screens)

**Date:** 2026-06-02  
**Status:** Awaiting user approval — do not implement until approved.

## Question

How should Kachuful handle the full round lifecycle (deal → call → play → round end → next round) on **one continuous game view**, without screen swaps, modal takeovers, or jarring layout changes — especially at trick end and round end?

## References

| Source | Pattern |
|--------|---------|
| [Hearts (shmemcat)](https://github.com/shmemcat/hearts) | Single-page real-time table; card dealing + trick animations on same canvas; score overlays; no route change between tricks. |
| [Hearts (awang9999)](https://github.com/awang9999/hearts-game) | Active-player highlight, leading-card glow, **inline trick-winner announcement** (not a new page); hand score vs total score separated in UI. |
| [Bisca multiplayer](https://github.com/n-alex-goncalves/BiscaMultiplayer) | Framer Motion: **cards animate toward trick winner**; spring physics; same table throughout. |
| [Spades Card (browser)](https://spades-card.com/) | HTML5 single view; responsive table; rejoin without leaving view; UI reachable while playing. |
| Teen Patti UI patterns | Fixed zones: **center = play**, **edges = players**, **overlays = non-blocking toasts**; snappy deal (~200–400ms), slower win moment (~800–1200ms). |

## Patterns to adopt

1. **One camera, one layout** — `Game.jsx` never unmounts the table shell; only inner table **phase** changes (deal / call / play / trick-resolve / round-score / between-rounds).
2. **Player seats fixed around ellipse** — you always at bottom; names/avatars don’t jump when phase changes.
3. **Mid-trick resolve (not last trick)** — ~1.0–1.2s on same felt: highlight winner card → optional slide toward winner seat → clear center → next lead. **No sidebar hide/show, no replacing entire table component.**
4. **Round end (last trick)** — extend mid-trick flow:
   - Same trick highlight (1.0s)
   - Cards **animate to center deck pile** (0.6–0.8s)
   - Deck shrinks / fades; **seat badges** appear: `Call X · Won Y · +Z pts` (round only)
   - Small action bar **inside table footer**: Host “Next round” + “Leave” for all
   - Hand area dims or hides until next deal
5. **No session totals during round** — sidebar shows call + tricks won only; cumulative score **only on final leaderboard** when session ends.
6. **Deal (next round)** — center shows deck stack → cards fly to seats one-by-one (stagger 80–120ms per card); then calling UI appears in sidebar (unchanged layout).
7. **State machine** — explicit phases in client (server truth unchanged):

```
idle → dealing → calling → playing → trick-resolve → [playing | round-collect → round-score → between-rounds → dealing]
```

8. **Reduced motion** — if `prefers-reduced-motion`, skip fly animations; show static badges only.

## Patterns to avoid

| Anti-pattern | Why it failed for us |
|--------------|----------------------|
| Full-page `RoundSummary` / `RoundEndReveal` route | User feels “new screen”; breaks immersion. |
| Swapping `GameTable` ↔ `TrickReveal` as separate components | Layout jump; hand disappears; annoying between tricks. |
| 3+ second blocking pause mid-round | Feels like broken loading. |
| Showing session total at round end | User wants round pts at seats only; totals at session end. |
| `tablePhase` driving sidebar mount/unmount | Causes flicker and “different app” feeling. |
| Auto-advancing to next round without host | Spec wants vote/host control (host-only OK for now). |

## Proposed flow (user experience)

### A. Start of round
1. Table stays visible; center shows **face-down deck** (3-card stack graphic).
2. Text: “Dealing…” — cards animate from deck to each seat (visual only; hands already on server).
3. Deck fades; sidebar shows **Call picker** when it’s your turn to call.
4. Sar badge stays top-center (unchanged).

### B. During play
1. You play from hand below table (always same position).
2. Played cards appear in **center trick area** (current behavior, keep).
3. Turn indicator: glow on **seat**, not a big banner swap.

### C. Mid-round trick ends
1. All played cards stay visible ~1s; winner seat + card get gold ring.
2. Optional: cards slide slightly toward winner (150px max).
3. Center clears; winner leads next trick. **Sidebar and hand stay.**

### D. Last trick / round ends
1. Same as (C) for ~1s.
2. All trick cards animate **into center deck** (merge animation).
3. Deck pulse, then disappears.
4. At each seat, small label fades in:  
   `Called 1 · Won 1 · +11 pts` (example; **no “total 22”**).
5. Table footer: **[Next round]** (host) · **[Leave session]** (all).
6. Non-host sees: “Waiting for host…” under buttons.
7. Screen **does not navigate**; URL stays `/game/:code`.

### E. Next round
1. Host taps Next round → seat badges fade out.
2. Center deck returns → deal animation → calling phase.
3. Mountain round / Sar updates in header only.

### F. Session end (out of scope for this pass, but aligned)
1. After last mountain round → navigate to **leaderboard** with full ranks + session totals.

## Open questions for user

1. **Mid-trick pause length** — OK with **~1.2s** highlight (not 3s)?
2. **Card fly to winner** on every trick, or **only on round-ending trick**?
3. **Deal animation** — full one-by-one deal for all cards, or simplified (deck flash + cards appear in hand)?
4. **Next round** — host-only button OK until voting is built?
5. **Leave session** during round-score — immediate exit, or confirm dialog?
6. **Spectators** — see seat badges but no hand; OK?

## Implementation scope (after approval)

### Phase 1 (MVP — fix annoyance)
- Remove any remaining full-layout swaps; single `GameTable` shell always mounted.
- Mid-trick: inline highlight only, **1.2s**, no `tablePhase` hiding hand/sidebar.
- Round end: seat badges + footer actions on same table; **hide session totals**.
- Files: `GameTable.jsx`, `Game.jsx`; delete dead phase logic.

### Phase 2 (polish)
- Framer Motion: cards → winner (mid-trick), cards → deck (round end).
- Deal animation from center deck.
- `prefers-reduced-motion` fallback.

### Phase 3 (later)
- Voting (next vs end session) on table footer instead of host-only.
- Sound/haptic on trick win (optional).

### Out of scope
- Firebase/server rule changes (reuse `lastTrickReveal`, `acknowledgeTrickReveal`, `advanceToNextRound`).
- Final leaderboard redesign.
- Lobby/join-request UX.

## Success criteria

- [ ] User cannot tell a “new screen” appeared at trick or round end.
- [ ] Hand + table + sidebar layout stable; only center + seat overlays change.
- [ ] Round points visible at seats; no session total until leaderboard.
- [ ] Mid-trick interrupt ≤ 1.5s before next play.
- [ ] Host can start next round without leaving table view.
